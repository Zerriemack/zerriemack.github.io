# Project Retrospective

## What went well
This project came together smoothly because I kept the structure simple and focused on the requirements. I built the pages first, then styled them, then went back and tightened the content. Having one stylesheet made it easy to keep the site consistent across pages, and once the navigation was in place I could move quickly without creating extra cleanup work later.

I also got more comfortable working in GitHub as the real home for the project. I am quicker at making updates, committing changes, and checking the live deployment through GitHub Pages. The docs folder helped me stay organized since the scope, plan, and retrospective were all in one place and easy to verify before submission.

Linking out to Milly Maker Tracker instead of trying to rebuild any of it inside the hub was the right move. It kept the hub focused on organization and presentation, and it stayed aligned with what I called out as out of scope.

## What went wrong
Nothing major broke, but I did notice two small friction points. First, I spent more time than expected making sure every link was correct across both the repository README and the site itself. It was not hard, but it forced me to slow down and double check every destination because one wrong character would cause a broken route.

Second, I caught myself polishing details early even though the project did not require advanced styling. I had to remind myself to finish the documentation and confirm the core navigation before I worried about minor visual tweaks.

## What I would do differently
If I were starting over, I would take a more courageous approach with the design. I played it safe so I could meet the requirements quickly, but I can see how this hub could become a real portfolio asset. Next time I would push my limits with a bolder layout, stronger visual hierarchy, and a more distinct personal style while still keeping the site clean and easy to navigate.

I would also set a tighter workflow for link verification. I would create a short checklist early and run the same test every time before pushing changes. That would make the final link check faster and reduce the chance of missing a small mistake.
